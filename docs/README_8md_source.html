3 # ECE590_Project
4 ## Project Goals
5 
6 The goals of this project are to port elma onto the Raspberry Pi and add multithreading functionality to elma.
7 
8 This will be tested by porting elma to the Raspberry Pi and using the sense hat to create a visual display that requires quick and uninterrupted scheduler.

38 ## Results
39 
40 Performance can be tested/measured with ./test/bin/multithread
41 
42 ```
43 //Process decloration
44 SlowSingle(string name) : Process(name, 1, SINGLE) {}
45 SlowSingle(string name) : Process(name, 1, MULTI) {}
46 
47  //Slow sleeps a large amount of time
48  void update() {
49  usleep(30000);
50  Slow_ADDIT++;
51  }
52 
53  //Fast simply adds 1 and returns
54  void update() {
55  Fast_ADDIT++;
56  }
57 
58  //populate schedualer
59  m.schedule(SLEEP, 20_ms)
60  .schedule(count, 3_ms)
61  .init()
62  .run(91_ms);
63 
64  //results
65  SINGLE Fast_ADDIT: 8 Slow_ADDIT : 3
66  MULTI Fast_ADDIT: 30 Slow_ADDIT : 4
67 ```
68 
69 The test file shows single threading is subject to delays under long or slow updates. Multi threading works as expected allowing for full amount of updates in the given runtime.
70 
71 Overall the results are very positive. The multithreading seems to work well with elma and has several valid use cases including client.h/client.cc. The Pi working example also works well, and uses multithreading (by choice not by necessity)
72 
73 ### Notes
74 
75 * Because the process are ditached timing becomes odd, a sleep command in the test file was needed to ensure all update process finished before the main program exited.
76 
77 * Originally, I attemped to add the multithreading in to the manager directly, but after many challanges, found it easier to place it in the process itself. Several different wrappers and reference libraries were used with no success. One advantage of placing mulithreading in the process directly is it allowed easier variable control and interaction with the manager.
